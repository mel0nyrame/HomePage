## 已实现功能

- 用户两步注册:图形验证码 + 邮箱验证码双校验
  - 第一步 `POST /api/user/register` 校验图形验证码、密码是否泄露、用户名/邮箱是否被占用,将待写入的 `UserEntity` 序列化后存 Redis(5 分钟过期),同时发送邮箱验证码
  - 第二步 `POST /api/user/email/verify` 校验邮箱验证码后,真正落库
  - `POST /api/user/email/retry` 重新发送邮箱验证码
- 用户/管理员登录,均要求图形验证码
- 管理员注册(需 `ROLE_ADMIN`)
- JWT 双 token 机制(`access_token` 30 分钟 + `refresh_token` 7 天)
  - `TokenService` 统一负责签发、旋转(rotation)、吊销、解析
  - `TokenStore` 把 token 落到 Redis,支持 access/refresh 校验、`jti → username` 反查、按用户名一键吊销
  - `JwtValidationFilter` 后置过滤器,校验 `type=access_token` + Redis 中 jti 仍存在(防止 refresh_token 误用、防吊销)
  - `/api/user/refresh`、`/api/user/logout`、`/api/user/logout/all` 等刷新 / 登出 / 全下线接口
- 基于角色的权限控制(`@PreAuthorize("hasRole('ADMIN')")`)
- 密码泄露检查(`HaveIBeenPwnedRestApiPasswordChecker` 集成)
- 图形验证码(滑动干扰图,Redis 存储,60 秒过期,base64 返回)
- 邮箱验证码(6 位数字,Redis 存储 5 分钟,Thymeleaf 模板渲染,异步发送)
- 全局异常处理、统一响应格式、Swagger UI、Actuator
- MyBatis-Plus 自动填充(`create_time` / `update_time`)
- 首次启动自动创建初始管理员(`DataInitializer`)

## 业务逻辑

### 一、JWT 双 token 机制

方案:**Spring Security 6 + OAuth2 Resource Server + Nimbus JWT + BCrypt + Redis(存储与吊销)**

---

#### 1. JWT 结构

`JwtUtil` 统一签发 access / refresh token,在标准 claim 之外多加了 `jti` 和 `type`:

| Claim | 值 | 说明 |
| --- | --- | --- |
| 算法 | HS256 | 对称加密,签发和验证用同一密钥(从 `jwt.secret` 读取) |
| `iss` | `"homepage"` | 签发者 |
| `iat` | 当前时间 | 签发时间 |
| `exp` | `iat` + 30 min(access) / 7 d(refresh) | 过期时间 |
| `sub` | 用户名 | 令牌主体 |
| `jti` | UUID 随机串 | 写入 Redis 用于吊销与反查 |
| `scope` | 权限列表 | 空格拼接,如 `USER`、`ADMIN` |
| `type` | `access_token` / `refresh_token` | token 类型,后置过滤器防混用 |

**常量定义**(`JwtConstants`):

```java
public static final Long JWT_ACCESS_TOKEN_EXPIRATION_TIME  = 30 * 60 * 1000L;            // 30 分钟
public static final Long JWT_REFRESH_TOKEN_EXPIRATION_TIME = 7 * 24 * 60 * 60 * 1000L;    // 7 天
```

---

#### 2. Token 在 Redis 中的存储设计

`RedisConstants` 与 `TokenStore` 配合,key 设计如下:

| Key | 类型 | Value | 用途 |
| --- | --- | --- | --- |
| `token:access:{jti}` | String | 完整 access_token 字符串 | 校验 access 是否仍有效 |
| `token:refresh:{jti}` | String | 完整 refresh_token 字符串 | 校验 refresh 是否仍有效 |
| `token:user:refresh:{username}` | Set\<jti\> | 该用户所有 refresh 会话 jti | 一键踢下线 |
| `token:refresh:index` | Hash | `jti → username` | refresh 接口根据 jti 反查 username,重新加载 UserDetails |

TTL:access / refresh 等同其 token 自身 TTL;用户会话集合的 TTL 比 refresh 略长 60 秒,避免边角问题。

---

#### 3. 签发 / 旋转 / 吊销流程

`TokenService` 是统一的 token 业务层,User 与 Admin 的 Service 都通过它操作 token。

**(1) 签发双 token 对(`issueTokenPair`)**

```
登录 / 刷新成功
  │
  ▼
jwtUtil.generateAccessToken(authentication)  → access_token(jti_a, type=access_token)
jwtUtil.generateRefreshToken(authentication) → refresh_token(jti_r, type=refresh_token)
  │
  ▼
tokenStore.saveAccess(jti_a, access_token,  30*60s)
tokenStore.saveRefresh(jti_r, username, refresh_token, 7*24*60*60s)
  │           ├─ token:refresh:{jti_r} = refresh_token
  │           ├─ token:refresh:index  HSET jti_r username
  │           └─ token:user:refresh:{username} SADD jti_r
  │
  ▼
返回 TokenDTO { accessToken, refreshToken }
```

**(2) 旋转(`rotate`)—— refresh_token 换新对,带 reuse-detection**

```
客户端 POST /api/{user|admin}/refresh
  Header: X-Refresh-Token: <refresh_token>
  │
  ▼
TokenService.rotate(refreshToken, userDetailsService)
  ├── 1. jwtDecoder.decode() 验签 + 验 exp(失败 → USER_TOKEN_INVALID)
  ├── 2. 校验 claim type == "refresh_token"(否则 USER_TOKEN_INVALID)
  ├── 3. jti = jwt.getId()
  ├── 4. 正常路径:tokenStore.isRefreshValid(jti, refreshToken) == true
  │        ├── tokenStore.revoke(jti)  // 吊销旧 refresh
  │        └── issueTokenPair(...)     // 重新签发新对
  │
  └── 5. 异常路径:旧 refresh 已失效但又出现 → 视为 token 复用
           ├── tokenStore.getUsernameByRefreshJti(jti) 反查 username
           ├── tokenStore.revokeAllByUsername(username)  // 踢掉该账号所有会话
           └── 抛 USER_TOKEN_INVALID
```

**(3) 吊销**

| 方法 | 作用 |
| --- | --- |
| `TokenService.revoke(accessJti)` | 当前会话登出:吊销传入 jti 的 access + refresh |
| `TokenService.revokeAll(username)` | 全下线:吊销该用户名下所有 refresh 会话及对应 access |

---

#### 4. 鉴权链路

```
请求到达(Header: Authorization: Bearer <access_token>)
  │
  ▼
① BearerTokenAuthenticationFilter(Spring Security 默认)
   ├── 提取 Bearer token
   │     └── 缺失/格式错 → RestAuthenticationEntryPoint → 401
   ├── NimbusJwtDecoder.decode(token) 验签 + 验 exp
   │     └── 失败 → 401
   ├── JwtAuthenticationConverter 把 scope claim 转 GrantedAuthority
   │     "USER"  → ROLE_USER
   │     "ADMIN" → ROLE_ADMIN
   └── 写入 SecurityContext
  │
  ▼
② JwtValidationFilter(自定义,挂载在 ① 之后)
   ├── 仅对已解析出 Jwt 主体的请求做后置校验
   ├── 校验 type claim == "access_token"(防止 refresh_token 被当 access_token 用)
   │     └── 失败 → 清空 SecurityContext + BearerTokenAuthenticationEntryPoint → 401
   └── 校验 tokenStore.isAccessValid(jti, token)(Redis 中仍存在且一致)
         └── 失败 → 同上,401
  │
  ▼
③ 业务 Controller
   可通过以下方式获取当前用户:
   - SecurityContextHolder.getContext().getAuthentication()
   - @AuthenticationPrincipal Jwt jwt
   │
   ▼
权限不足 → RestAccessDeniedHandler → 403
```

**SecurityConfig 实际配置:**

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity          // 启用 @PreAuthorize 方法级权限注解
public class SecurityConfig {

    private final RestAuthenticationEntryPoint restAuthenticationEntryPoint;
    private final RestAccessDeniedHandler restAccessDeniedHandler;
    private final JwtValidationFilter jwtValidationFilter;
    private final String[] requestMatchers;   // 注入白名单,见配置

    public SecurityConfig(RestAuthenticationEntryPoint restAuthenticationEntryPoint,
                          RestAccessDeniedHandler restAccessDeniedHandler,
                          JwtValidationFilter jwtValidationFilter) {
        this.restAuthenticationEntryPoint = restAuthenticationEntryPoint;
        this.restAccessDeniedHandler = restAccessDeniedHandler;
        this.jwtValidationFilter = jwtValidationFilter;
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public JwtAuthenticationConverter jwtAuthenticationConverter() {
        JwtGrantedAuthoritiesConverter grantedAuthoritiesConverter = new JwtGrantedAuthoritiesConverter();
        grantedAuthoritiesConverter.setAuthoritiesClaimName("scope");
        grantedAuthoritiesConverter.setAuthorityPrefix("ROLE_");

        JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
        converter.setJwtGrantedAuthoritiesConverter(grantedAuthoritiesConverter);
        return converter;
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http,
                                                   JwtAuthenticationConverter jwtAuthenticationConverter) throws Exception {
        http
                .csrf(AbstractHttpConfigurer::disable)
                .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
                .oauth2ResourceServer(oauth2 -> oauth2
                        .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtAuthenticationConverter)))
                // 在默认 BearerToken 过滤器之后做 type + Redis 校验
                .addFilterAfter(jwtValidationFilter, BearerTokenAuthenticationFilter.class)
                .exceptionHandling(ex -> ex
                        .authenticationEntryPoint(restAuthenticationEntryPoint)
                        .accessDeniedHandler(restAccessDeniedHandler))
                .headers(headers -> headers
                        .frameOptions(HeadersConfigurer.FrameOptionsConfig::deny)
                        .contentTypeOptions(Customizer.withDefaults())
                        .httpStrictTransportSecurity(hsts -> hsts
                                .includeSubDomains(true)
                                .maxAgeInSeconds(31536000)))
                .authorizeHttpRequests(auth -> auth
                        .requestMatchers(requestMatchers).permitAll()
                        .anyRequest().authenticated());
        return http.build();
    }

    // 两个独立的 AuthenticationManager,各自绑定专用的 UserDetailsService
    @Primary
    @Bean("userAuthenticationManager")
    public AuthenticationManager userAuthenticationManager(UserDetailsServiceImpl userDetailsService,
                                                           PasswordEncoder passwordEncoder) {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider(userDetailsService);
        provider.setPasswordEncoder(passwordEncoder);
        return new ProviderManager(provider);
    }

    @Bean("adminAuthenticationManager")
    public AuthenticationManager adminAuthenticationManager(AdminUserDetailsServiceImpl adminDetailsService,
                                                            PasswordEncoder passwordEncoder) {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider(adminDetailsService);
        provider.setPasswordEncoder(passwordEncoder);
        return new ProviderManager(provider);
    }

    @Bean
    public CompromisedPasswordChecker compromisedPasswordChecker() {
        return new HaveIBeenPwnedRestApiPasswordChecker();
    }
}
```

**JWT 编解码配置(JwtConfig,密钥从配置注入):**

```java
@Configuration
public class JwtConfig {

    @Value("${jwt.secret}")
    private String jwtSecret;

    @Bean
    public JwtEncoder jwtEncoder() {
        SecretKey key = new SecretKeySpec(jwtSecret.getBytes(StandardCharsets.UTF_8), "HmacSHA256");
        JWKSource<SecurityContext> jwks = new ImmutableSecret<>(key);
        return new NimbusJwtEncoder(jwks);
    }

    @Bean
    public JwtDecoder jwtDecoder() {
        return NimbusJwtDecoder
                .withSecretKey(new SecretKeySpec(jwtSecret.getBytes(StandardCharsets.UTF_8), "HmacSHA256"))
                .build();
    }
}
```

### 二、用户登录、注册、邮箱验证

`LoginDTO` / `RegisterDTO` 均实现 `CaptchaAware`,登录和注册都要先过图形验证码。

---

#### 1. 用户登录

`POST /api/user/login`

```
客户端提交 { account, password, captchaId, captcha }
  │
  ▼
UserController.login()
  │
  ▼
UserServiceImpl.login()
  ├── 1. RedisUtil.verifyCaptcha(loginDTO)   // 图形验证码(Lua 原子化 GET+DEL)
  │     └── 错误/过期 → USER_VERIFY_CODE_ERROR / USER_VERIFY_CODE_EXPIRED
  ├── 2. userAuthenticationManager.authenticate(token)
  │     ├── DaoAuthenticationProvider 回调 UserDetailsServiceImpl.loadUserByUsername()
  │     │     └── 查询 UserEntity,包装为 HomepageUserDetails(authorities = "USER")
  │     ├── 密码错误 → BadCredentialsException → USER_PASSWORD_ERROR
  │     ├── 账号禁用 → DisabledException → USER_ACCOUNT_DISABLED
  │     └── 认证通过 → 返回 Authentication
  └── 3. TokenService.issueTokenPair(authentication) → 返回 { accessToken, refreshToken }
```

---

#### 2. 用户注册(两步)

**第一步:`POST /api/user/register`**

```
客户端提交 { nickname, username, email, password, captchaId, captcha }
  │
  ▼
UserServiceImpl.register()
  ├── 1. RedisUtil.verifyCaptcha(registerDTO)               // 图形验证码
  ├── 2. CompromisedPasswordChecker.check(password)         // 密码泄露检查
  │     └── isCompromised() → USER_PASSWORD_LEAKED
  ├── 3. lambdaQuery().eq(username) | .eq(email) → 已存在:
  │     username 占 → USER_ALREADY_EXIST(1002)
  │     email 占    → USER_EMAIL_EXIST(1009)
  ├── 4. BCryptPasswordEncoder.encode(明文) → 密文
  ├── 5. 构造 UserEntity(enabled=1, authorities="USER"),序列化为 JSON
  │     └── redisTemplate.set("user:" + email, json, 5 min)   // 待写入缓存
  └── 6. MailUtil.sendEmail(email) 异步发送 6 位邮箱验证码,5 分钟过期
```

**第二步:`POST /api/user/email/verify`**

```
客户端提交 { email, captcha }
  │
  ▼
UserServiceImpl.verifyEmail()
  ├── 1. RedisUtil.verifyEmailCaptcha(email, captcha)       // 邮箱验证码(Lua 原子化)
  │     └── 错误 → USER_VERIFY_CODE_ERROR
  │     └── 过期/不存在 → USER_VERIFY_CODE_EXPIRED
  ├── 2. redisTemplate.getAndDelete("user:" + email)
  │     └── 取不到(已被消费/超时) → USER_VERIFY_CODE_EXPIRED
  └── 3. JSONUtil.toBean → UserEntity → this.save(user) 真入库
```

**重发验证码:`POST /api/user/email/retry`**

```
MailUtil.sendEmail(email) 异步重发,5 分钟过期
```

#### 1. 管理员登录

  │     ├── DaoAuthenticationProvider 回调 AdminUserDetailsServiceImpl.loadUserByUsername()

  │     ├── 账号禁用 → DisabledException → USER_ACCOUNT_DISABLED

#### 2. 管理员注册(需 ADMIN 权限)

`POST /api/admin/register`  
权限要求:`@PreAuthorize("hasRole('ADMIN')")`  

```
客户端提交 { nickname, account, password, captchaId, captcha }(Header 需携带 ADMIN 的 access_token)
  │
  ▼
AdminServiceImpl.register()(@Transactional)
  ├── 1. RedisUtil.verifyCaptcha(adminRegisterDTO)         // 图形验证码
  ├── 2. CompromisedPasswordChecker.check(password)        // 密码泄露检查
  ├── 3. lambdaQuery().eq(AdminEntity::getAccount, account).exists()
  │     └── 已存在 → USER_ALREADY_EXIST
  ├── 4. BCryptPasswordEncoder.encode(明文) → 密文
  └── 5. this.save(adminEntity) 入库(enabled=1, authorities="ADMIN")
```

---

#### 3. 管理员刷新 / 登出 / 全下线

与用户同名接口等价,`/api/admin/refresh|logout|logout/all`,行为一致。

#### 4. 角色权限机制

- User 登录 → JWT scope = `USER` → `JwtAuthenticationConverter` → `ROLE_USER`
- Admin 登录 → JWT scope = `ADMIN` → `JwtAuthenticationConverter` → `ROLE_ADMIN`
- 授权注解:`@PreAuthorize("hasRole('ADMIN')")` 检查 `ROLE_ADMIN`
- `@EnableMethodSecurity` 必须显式开启,否则 `@PreAuthorize` 不生效

### 四、图形验证码 / 邮箱验证码

#### 1. 图形验证码

`POST /api/captcha/`(无需登录)

```
CaptchaServiceImpl.captcha()
  ├── Hutool ShearCaptcha 250×50, 5 位字符,6 像素干扰线
  ├── 验证码 code → redisTemplate.set("auth:captcha:" + uuid, code, 60s)
  └── 返回 { captchaId(uuid), image("data:image/png;base64,...") }
```

校验:`RedisUtil.verifyCaptcha(...)` 用 Lua 脚本 **GET + DEL 原子化** 读出并立即删除,防并发重放;`MessageDigest.isEqual` 常量时间比较防时序攻击。

#### 2. 邮箱验证码

```
MailUtil.sendEmail(email)(@Async, CompletableFuture<Void>)
  ├── 6 位数字验证码
  ├── redisTemplate.set("email:captcha:" + email, captcha, 5 min)
  ├── Thymeleaf 渲染 "EmailVerificationCode.html"(逐字符显示)
  └── JavaMailSender 异步发送
```

### 五、实体与 Spring Security 解耦设计

实体类(`AdminEntity`、`UserEntity`)是纯数据库映射 POJO,通过独立的 `UserDetailsService` + 包装类对接 Spring Security:

```
AdminEntity(纯 POJO, @TableName("homepage_admin"))
    │
    └── AdminUserDetails(实现 UserDetails, 包装 AdminEntity)
            ↑ 由 AdminUserDetailsServiceImpl.loadUserByUsername() 构造

UserEntity(纯 POJO, @TableName("homepage_user"))
    │
    └── HomepageUserDetails(实现 UserDetails, 包装 UserEntity)
            ↑ 由 UserDetailsServiceImpl.loadUserByUsername() 构造
```

`AdminUserDetails` / `HomepageUserDetails` 均实现:`getAuthorities()` 读实体 `authorities` 字段并用 `commaSeparatedStringToAuthorityList` 转成 `GrantedAuthority`;`isEnabled()` 读 `enabled == 1`;账户过期/锁定/凭证过期均返回 `true`(暂未启用)。

## GitHub 工作流与脚本

`.github/` 目录下有两类自动化:**GitHub Actions CI**(`workflows/`,在 GitHub 端跑)和**本地同步脚本**(`scripts/`,在开发者本机跑)。两者围绕同一份产物 `CHANGELOG.md` 协作:CI 端生成,本地定时拉取。

整体数据流:

```
开发者 push → main 分支
    │
    ▼
[CI] .github/workflows/changelog.yml
    │  触发 .github/scripts/gen-changelog.sh
    ▼
CI commit + push CHANGELOG.md 回 main(committer = github-actions[bot])
    │
    ▼
[本机] 定时任务每 10 分钟跑一次
    │  .github/scripts/sync-changelog.sh
    │  └─ 拉取 origin/main:CHANGELOG.md,内容有差异才覆盖本地
    ▼
本机 CHANGELOG.md 与 main 保持一致
```

---

### 一、GitHub Actions 工作流

#### 1. `workflows/changelog.yml` — 自动生成 CHANGELOG

| 项 | 值 |
| --- | --- |
| 触发 | `push` 到 `main` 分支 |
| 运行环境 | `ubuntu-latest` |
| 权限 | `contents: write` |
| 守卫 | `if: github.actor != 'github-actions[bot]'` —— 避免本工作流自身的 commit 触发再生成 |
| 步骤 | ① checkout(完整历史) → ② 执行 `bash .github/scripts/gen-changelog.sh` → ③ 提交变更(commit message `docs: 更新 CHANGELOG.md [skip ci]`) |

**关键点:**

- `fetch-depth: 0` 拉全历史,`gen-changelog.sh` 才能遍历所有提交
- 用 `git diff --cached --quiet` 判断有无差异,无差异不产生空 commit
- commit message 带 `[skip ci]`,防止再次触发 changelog 工作流

#### 2. `workflows/ggshield.yml` — GitGuardian 密钥扫描

| 项 | 值 |
| --- | --- |
| 触发 | `push` 到 `main` 或 `dev*` 分支;`pull_request` 到 `main` |
| Action | `GitGuardian/ggshield-action@v1` |
| 所需 Secret | `GITGUARDIAN_API_KEY`(`GITHUB_TOKEN` 自动可用) |
| 守卫 | `if: github.actor != 'github-actions[bot]'`,避免扫描自己的 commit |

防泄漏:任何 push 或合入 main 的 PR 都会过一遍 GitGuardian,命中则阻断。

---

### 二、`scripts/gen-changelog.sh` — CI 端 CHANGELOG 生成器

遍历 `git log main` 的所有提交(从最早到最新),对每个 commit 输出以下结构:

```markdown
---

## YYYY-MM-DD — <commit subject>

**`<short_hash>`** | <author>

### 改动文件

| 文件 | + | - |
|------|---|---|
| `<path>` | <added> | <deleted> |

### 改动位置

- <从 `git diff-tree -p` 的 hunk 头解析出的方法/类/上下文行>
```

**特殊提交处理:**

| 类型 | 判定 | 输出 |
| --- | --- | --- |
| 合并提交 | `parent_count > 1` | 标题后追加 `*合并提交,变更细节见各子提交。*`,跳过文件 / 位置列表 |
| 根提交(初始) | `parent_count == 0` | 改动位置输出 `*初始提交,全部新增。*` |

**改动文件表:** 用 `git diff-tree --numstat -r <hash>` 拿到 `added  deleted  file`,直接列表;若 numstat 为空(理论上不该发生),输出 `*无文件变更*`。

**改动位置列表:** 用 `git diff-tree -p` 解析 `@@` hunk 头,从中提取上下文行(`@@ -A,B +C,D @@ <context>` 中的 `<context>` 段),过滤掉 `import` / `package` / 单字母 `xxx:` 这样的行后去重输出。

**末尾签名:** `*自动生成于 YYYY-MM-DD HH:MM:SS UTC*`(用 `date -u` 取 UTC)。

`set -euo pipefail` 严格模式,任何一步失败立刻终止。

---

### 三、本地 CHANGELOG 同步

`CHANGELOG.md` 由 CI 推到 main,本机工作树里的副本会过时。同步方案是每 10 分钟拉取一次远端版本,内容有差异才覆盖。

#### 1. `scripts/sync-changelog.sh` — 核心同步逻辑

```bash
#!/bin/bash
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
cd "$REPO_ROOT"

git fetch origin main

LOCAL_HASH=$(git hash-object CHANGELOG.md 2>/dev/null || echo "")
REMOTE_HASH=$(git hash-object origin/main:CHANGELOG.md 2>/dev/null || echo "")

if [ "$LOCAL_HASH" = "$REMOTE_HASH" ]; then
    echo "CHANGELOG.md 已是最新,无需更新。"
    exit 0
fi

git checkout origin/main -- CHANGELOG.md
echo "CHANGELOG.md 已同步到远程 main 最新版本。"
```

**关键点:**

- 用 `git hash-object` 比对 blob hash,而不是直接看 mtime / 大小,避免内容相同但元数据不同造成的误更新
- 走 `git checkout origin/main -- CHANGELOG.md`,只覆盖这一个文件,不污染工作树其他内容
- `|| echo ""` 兜底:本地或远端没有 `CHANGELOG.md` 时,空串比较会判定为"有差异",后续 checkout 会创建/覆盖

#### 2. `scripts/sync-changelog.bat` — Windows 包装

```bat
@echo off
"C:\Program Files\Git\usr\bin\bash.exe" -l -c "cd /c/Users/klei/Desktop/HomePage && bash .github/scripts/sync-changelog.sh"
```

直接调 Git Bash 执行同步脚本。**注意:此文件硬编码了 Windows 路径 `/c/Users/klei/Desktop/HomePage`,使用前需要改成实际的项目路径。**

#### 3. `scripts/install-sync.sh` — 跨平台一键安装

根据 `uname -s` 自动选择定时任务方案:

| 系统 | 注册方式 | 频率 | 任务名 / 入口 |
| --- | --- | --- | --- |
| macOS(`Darwin`) | `launchd` | `StartInterval = 600`(10 分钟) | `~/Library/LaunchAgents/com.homepage.sync-changelog.plist` |
| Linux | `crontab` | `*/10 * * * *` | `bash <repo>/.github/scripts/sync-changelog.sh` |
| Windows Git Bash(`MINGW*`/`MSYS*`/`CYGWIN*`) | `schtasks` | `//sc minute //mo 10` | 任务名 `Sync Changelog`,执行 `sync-changelog.bat` |

**macOS 分支额外做的事:**

1. 读取 `scripts/com.homepage.sync-changelog.plist` 模板
2. `sed` 把里面的占位符 `/Users/REPLACE_USERNAME/Desktop/HomePage` 替换成实际仓库根路径
3. 写入 `~/Library/LaunchAgents/`
4. 先 `launchctl unload`(若已加载),再 `launchctl load` —— 保证幂等

**Linux 分支:** 先 `crontab -l | grep -v` 去掉旧条目,再追加新行,避免重复。

**Windows 分支:** 优先直接传 Unix 路径的 bat;失败时回退到 `cygpath -w` 转换 Windows 路径。`//f` 强制覆盖同名任务。

**入口校验:** `set -euo pipefail` 严格模式,未知系统 `exit 1`,不会留下半成品配置。

**使用方式:**

```bash
bash .github/scripts/install-sync.sh   # 安装
bash .github/scripts/sync-changelog.sh  # 手动跑一次验证
```

#### 4. `scripts/com.homepage.sync-changelog.plist` — macOS launchd 模板

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist ...>
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.homepage.sync-changelog</string>
    <key>ProgramArguments</key>
    <array>
        <string>/bin/bash</string>
        <string>/Users/REPLACE_USERNAME/Desktop/HomePage/scripts/sync-changelog.sh</string>
    </array>
    <key>StartInterval</key>
    <integer>600</integer>
    <key>RunAtLoad</key>
    <false/>
    <key>StandardOutPath</key>
    <string>/tmp/com.homepage.sync-changelog.stdout.log</string>
    <key>StandardErrorPath</key>
    <string>/tmp/com.homepage.sync-changelog.stderr.log</string>
</dict>
</plist>
```

**注意:模板里的 `ProgramArguments` 路径是 `/Users/REPLACE_USERNAME/Desktop/HomePage/scripts/sync-changelog.sh`(注意是 `scripts/` 而非 `.github/scripts/`),实际项目里 sync-changelog.sh 在 `.github/scripts/` 下 —— 这与 `install-sync.sh` 中 `sed` 替换的占位符 `/Users/REPLACE_USERNAME/Desktop/HomePage` 对应的目标路径不一致,使用前需手工修正 plist 模板,或修改 `install-sync.sh` 的 sed 模式。**

| Key | 值 | 含义 |
| --- | --- | --- |
| `Label` | `com.homepage.sync-changelog` | launchd 任务唯一标识 |
| `ProgramArguments` | `["/bin/bash", "<repo>/scripts/sync-changelog.sh"]` | 实际执行的命令(注意路径以 `scripts/` 开头) |
| `StartInterval` | `600` | 每 600 秒跑一次 = 10 分钟 |
| `RunAtLoad` | `false` | launchd 加载时不立即跑,等下一个周期 |
| `StandardOutPath` | `/tmp/com.homepage.sync-changelog.stdout.log` | 标准输出日志 |
| `StandardErrorPath` | `/tmp/com.homepage.sync-changelog.stderr.log` | 错误输出日志 |

**常用命令:**

```bash
launchctl list | grep homepage            # 查看任务
launchctl unload ~/Library/LaunchAgents/com.homepage.sync-changelog.plist   # 卸载
launchctl load   ~/Library/LaunchAgents/com.homepage.sync-changelog.plist   # 加载
tail -f /tmp/com.homepage.sync-changelog.stdout.log                          # 跟踪日志
```
